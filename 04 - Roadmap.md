# Roadmap

> *"This roadmap is competency-based, not technology-based."*

The purpose of this roadmap is not to learn every framework.

The purpose is to progressively develop the knowledge, skills, and engineering judgment required to become a highly capable Senior, Staff, and eventually Principal Software Engineer.

Technologies will evolve.

Engineering competencies remain valuable throughout an entire career.

---

# Engineering Growth Model

Learning progresses through six major stages.

Each stage builds upon the previous one.

Progress is determined by demonstrated understanding and practical application rather than by time spent studying.

```text
Level 1
Engineering Foundations

↓

Level 2
Software Construction

↓

Level 3
Software Architecture

↓

Level 4
Distributed Systems

↓

Level 5
Cloud & Platform Engineering

↓

Level 6
Technical Leadership
```

A level is considered complete only when its competencies have been demonstrated through projects, discussions, and practical application.

# Level 1 — Engineering Foundations

## Goal

Develop a deep understanding of the principles that underpin all software engineering.

### Competencies

* Advanced JavaScript
* Advanced TypeScript
* Data Structures
* Algorithms
* Networking Fundamentals
* Operating Systems
* Git
* Linux
* Clean Code
* Refactoring
* Design Principles

### Recommended Books

* You Don't Know JS
* Effective TypeScript
* Clean Code
* Refactoring
* The Pragmatic Programmer

### Practical Projects

* Build a CLI application.
* Implement common data structures.
* Create a small HTTP server.
* Refactor an existing project.
* Implement design patterns in TypeScript.

### Mastery Criteria

You can:

* Explain concepts clearly.
* Write maintainable code.
* Identify code smells.
* Refactor confidently.
* Choose appropriate data structures.

## AI-Assisted Development

Learn to use AI responsibly.

Use AI to:

* Explore ideas.
* Review code.
* Learn concepts.
* Challenge assumptions.
* Accelerate experimentation.

Do not use AI as a substitute for thinking.

Every competency should produce at least one of the following:

* Production-ready project
* Technical article
* Architecture document
* ADR
* Presentation
* Open Source contribution
* Teaching session
* Benchmark
* Code Review
* Research summary

---

# Progress Evaluation

Progress is evaluated through evidence.

Examples include:

* Completed projects
* Architecture diagrams
* Code reviews
* Technical discussions
* Book notes
* Interview simulations
* Written documentation

Certificates alone are not considered evidence of mastery.

---

# Guiding Rule

Do not move to the next competency simply because a chapter has been completed.

Move forward only when the current competency can be:

* Explained.
* Applied.
* Defended.
* Taught.
* Improved.

Mastery is demonstrated through action, not completion.

---

# Final Goal

The destination is not to become an expert in a specific framework.

The destination is to become the kind of engineer who can confidently learn, evaluate, design, build, and evolve software systems regardless of the technologies involved.

This roadmap is intended to guide that lifelong journey.
